>_ENGINEERING REPORT

elisprs is an Emacs Lisp runtime in Rust that runs .el outside Emacs, built to become a fusevm frontend. This report describes the architecture, the reuse/own split with rust_lisp, the current state of the tree, and the dependency posture. The project is at milestone 1; numbers below are facts about the design, not aspirational metrics.

Summary

Emacs Lisp has historically been inseparable from its host editor. elisprs separates the two: it runs .el as standalone programs and is built to become the fifth consumer of fusevm — after zshrs, stryke, awkrs, and vimlrs.

Milestone 1 is a self-contained tree-walk evaluator: an elisp-correct reader, a Lisp-2 obarray with dynamic binding, 21 special forms with macros, and a ~65-subr standard library, all over the rust_lisp value model. The fusevm lowering, AOT path, LSP/DAP servers, and the cons-model upgrade (dotted pairs, setcar) are the milestone-2 work.

The reuse / own split

Emacs Lisp is a Lisp-2 (separate value and function cells per symbol) and is, by default, dynamically scoped. Those two facts are the personality of the language, and no general-purpose embeddable Lisp provides them. So the crate is split cleanly:

Reused — data model

rust_lisp (MIT) supplies Value / List / Symbol. Function objects ride in Value::Foreign(Rc<dyn Any>) so elisprs adds closures, macros, and subrs without forking the enum.

Own — the reader

rust_lisp's number-greedy tokenizer splits 1+/1- and lacks #', ?c, and dotted pairs. reader.rs is a small elisp-correct reader emitting rust_lisp Values.

Own — the obarray

Every symbol gets a value cell and a function cell; symbol lookup follows function-cell aliases. rust_lisp's single-namespace lexical eval is not used.

Own — dynamic binding

let/let* and lambda parameters bind through a specstack that saves and restores prior value cells — classic elisp dynamic scope.

The path to fusevm

Milestone 1 interprets; milestone 2 compiles. The seam is compiler.rs::lower, mirroring strykelang/strykelang/fusevm_native.rs:

.el  →  reader  →  Value  →  eval (tree-walk)            [milestone 1]
                          ↘  compiler.rs → fusevm::Chunk  [milestone 2]  →  fusevm VM + Cranelift JIT / AOT

One invasive core change

Add Value::{Cons, Symbol} to fusevm/src/value.rs and a dynamic-binding stack to the VM struct — the other frontends never needed dynamic scope or Lisp cells.

Extension ops

Reserve an elisp Op::Extended(id, arg) range and register a handler via vm.set_extension_handler(...) for quote / funcall / special-var bind / cons navigation.

Sub-chunks for lambdas

Each top-level form lowers into a ChunkBuilder; lambda bodies become sub-chunks. Subrs bind through vm.register_builtin(...).

JIT + AOT for free

Once lowered, elisp inherits fusevm's three-tier Cranelift JIT and aot::compile_object — the same way vimlrs, zshrs, and awkrs do.

Dependency posture

Milestone 1 is deliberately dependency-light — it builds offline with a single direct dependency. Milestone 2 adds fusevm (and, with the servers, lsp-server / lsp-types).

CrateRole
rust_lispS-expression value model — Value / List / Symbol (MIT); i64/f64 features for elisp-shaped numbers
fusevm (milestone 2)Language-agnostic bytecode VM + three-tier Cranelift JIT + AOT
lsp-server / lsp-types (planned)LSP transport + protocol types for --lsp

Honesty & longevity

Loud failures

Unsupported syntax (dotted pairs, backquote) errors at read time rather than silently misreading a config — the failure mode you want from a config language.

Real elisp semantics

A genuine Lisp-2 with dynamic binding, so behaviour tracks Emacs Lisp rather than a Scheme approximation.

Standalone crate

Milestone 1 builds on its own with one dependency; the fusevm path-dependency is added only when lowering begins.

Tested

cargo test covers the reader (unit) and end-to-end evaluation, including recursion, macros, and error handling.
